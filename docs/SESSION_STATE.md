## Зроблено

**Промпт 1 — тенанти і станції** (міграція `0010`, 74 тести в `test_operator_isolation.py`)

- `operators`, `operator_stations`, `operator_sessions`, `operator_payout_ledger` — у кожної `operator_id`
- Ізоляція тримається на трьох механізмах: `operator_id` першим аргументом і у `WHERE` кожної функції; `create_session` бере `operator_id` з підзапиту по станції, а не з аргументу; композитний FK `(station_id, operator_id)`
- Балансу оператора як колонки немає — `SUM` журналу, щоб не повторити розсинхрон `users.balance` ↔ `kw_transactions`
- Комісія в `Decimal` з `ROUND_HALF_UP` (`round()` по float зрізав копійку: 112.5 × 5% давало 5.62 замість 5.63)
- Ідемпотентність нарахувань — частковий унікальний індекс `uq_ledger_session_income`

**Промпт 2a — еквайринг оператора** (міграція `0011`, 32 тести в `test_operator_payments.py`)

- Інвойс створюється токеном мерчанта-**оператора**; гроші йдуть напряму йому, ми лише софт
- **Webhook не вірить власному тілу** — статус перепитується в банку токеном оператора. Причина: інвойси різних мерчантів підписані різними ключами, тож глобальна ECDSA-перевірка з `app/api/payments.py` для них не працює в принципі
- Невідомий `invoiceId` → тихий 200 без подробиць, щоб ендпоінт не став оракулом
- Токени операторів шифруються Fernet на `ENCRYPTION_KEY`; без ключа застосунок стартує, але видає гучний WARNING
- Окрема таблиця `operator_payments`, бо водій не реєструється, а `payments.user_id` — `NOT NULL`

**Промпт 2b — QR-флоу водія** (43 тести в `test_driver_qr.py`)

- `GET /s/{qr_slug}` → вибір суми → `POST /start` → інвойс → редірект у Monobank → webhook → пуш оператору «Оплачено, увімкніть станцію» → кнопка → сесія `charging`
- Водій не реєструється: єдиний ключ — `qr_slug`. Чек звіряє, що сесія належить саме цій станції
- Сторінки без жодного зовнішнього ресурсу (водій буває в підземному паркінгу); оновлення чека через `meta refresh`, не JS
- Оплату приймає лише оператор зі статусом `active`
