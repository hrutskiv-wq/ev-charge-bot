# 🖥️ Спільна Інфраструктура — розширення існуючого сервера ev-charge-bot

*База: DigitalOcean droplet, `/opt/ev-charge-bot`, Docker Compose (Postgres 16 + Redis 7 + контейнер bot). Нові проєкти підселяються сюди, а не на новий VPS.*

---

## Принцип

Сервер уже є, оплачується і працює (ev-charge-bot з P2P-модулем). ФОП Податковий Асистент і AI-Продавець — окремі застосунки, які **підключаються до спільних Postgres/Redis** через docker-мережу. Нуль нових серверів на старті.

## Цільова структура

```
DigitalOcean droplet
│
├── /opt/infra/                    ← НОВЕ: спільний шар
│   └── docker-compose.yml         Caddy (reverse proxy + авто-HTTPS)
│         ├── api.evolt.ua         → контейнер bot (webhooks, OCPI, PWA)
│         ├── fop.вашдомен.ua      → контейнер fop
│         └── shop.вашдомен.ua     → контейнер shop
│
├── /opt/ev-charge-bot/            ← існує; його postgres і redis
│   └── docker-compose.yml            стають СПІЛЬНИМИ:
│         postgres: + бази fop_db, shop_db (до існуючої)
│         redis:    logical DB 0 = evolt (FSM вже тут),
│                   DB 1 = fop (Celery), DB 2 = shop (черги)
│         усі сервіси → зовнішня docker-мережа infra_net
│
├── /opt/fop/    docker-compose.yml (тільки контейнер застосунку)
└── /opt/shop/   docker-compose.yml (тільки контейнер застосунку)
```

**Чому спільний Postgres, а не три:** одна БД-інстанція на 8 ГБ droplet тримає три MVP легко; окремі бази = окремі бекапи і простий переїзд будь-якого проєкту (`pg_dump → restore → DNS`) коли виросте. Дублювати три Postgres-контейнери на одному дроплеті — марна витрата RAM.

**Обережно з існуючим прод:** ev-charge-bot — живий сервіс з грошима. Будь-які зміни його compose-файлу — через гілку, з бекапом БД перед застосуванням (`scripts/backup_postgres.sh` вже є в проєкті).

## Економіка

| Стаття | Було б (окремі сервери) | Стало |
|---|---|---|
| Нові VPS | 2 × $25–50/міс | **$0** |
| Апгрейд droplet (якщо знадобиться) | — | +$12–24/міс (наступний тариф) |
| Бекапи/моніторинг | ×3 налаштування | одне, вже частково є |

У фінмоделях ФОП і Shop частка сервера ~$10–15/міс — це резерв під апгрейд droplet, не нова оренда.

**Коли апгрейдити/розселяти:** RAM стабільно > 80% або CPU > 70% у піки → спершу апгрейд droplet, потім винесення найважчого проєкту. Піки не збігаються: у зарядок — вечір, у ФОП — дні податкових дедлайнів, у Shop — рівномірно. Це на руку співмешканню.

## Промпт для Claude Code — міграція на спільний шар

**Виконати ОДИН раз, ПЕРЕД стартом ФОП/Shop. Агенту дати доступ до сервера і PROJECT_CONTEXT.md:**

> Обережно розшир існуючу інфраструктуру ev-charge-bot (/opt/ev-charge-bot, живий прод — перед змінами зроби бекап через scripts/backup_postgres.sh і покажи мені план дій до виконання). Кроки: (1) створи зовнішню docker-мережу infra_net; підключи до неї існуючі сервіси postgres, redis, bot у docker-compose.yml проєкту, не змінюючи їхні дані й томи. (2) Створи /opt/infra/docker-compose.yml з Caddy: reverse proxy з авто-HTTPS; маршрути: існуючий домен API → контейнер bot (порт як зараз), fop.<домен> → fop:8000, shop.<домен> → shop:8000; Caddy теж у infra_net. Якщо зараз порти bot опубліковані напряму — переведи публікацію 80/443 на Caddy, а bot лиши тільки в мережі. (3) Створи в існуючому Postgres бази fop_db і shop_db з окремими користувачами (паролі з /opt/infra/.env), розширення за потреби ставитимуть самі проєкти. (4) Розшир scripts/backup_postgres.sh: dump УСІХ баз (evolt + fop_db + shop_db), ротація як зараз. (5) Онови Uptime-моніторинг: додай перевірки /health для всіх трьох доменів (у ev-charge-bot ендпоінт /health вже є). (6) Перевір hardening: ufw 22/80/443, fail2ban, SSH без пароля — чого немає, додай. (7) README у /opt/infra з планом відкату кожного кроку.

**Після виконання** у стартових промптах ФОП і Shop діє фраза: *«Підключайся до спільних Postgres/Redis у мережі infra_net (бази fop_db / shop_db, Redis logical DB 1 / 2), не створюй власних»*.

## Чек-лист руками

1. Направити A-записи fop.<домен> і shop.<домен> на IP droplet
2. Виконати промпт вище; **перевірити, що ev-charge-bot піднявся** (`/health`, тестова команда в боті) одразу після кожного кроку
3. Тестове відновлення з бекапу на локальній машині (бекап без перевіреного restore не існує)
4. Вирішити зовнішнє сховище бекапів (DO Spaces/S3) — у backup-скрипті місце під один рядок `rclone`/`aws s3 cp` вже передбачено
5. Занести нові домени в Uptime-алерти в Telegram
