# План: OCPP (Промпт 3) + kWh-гаманець

> План/рішення. Дата: 2026-07-23. Доповнює `kwh-wallet-design.md` і
> `billing-commission-models.md` конкретними рішеннями та послідовністю.

## Велика картина
Ціль — **повна метрована модель**: на станції водій обирає, як платити —
**купленими кВт (баланс)** або **грн за фактично спожиті кВт**, і **ціна
формується ПІСЛЯ сесії** за реальним споживанням. Щоб рахувати спожите, бот
має **отримувати від зарядки виміряні кВт·год** → це вимагає **OCPP** (або
API розумної розетки з лічильником). Тому послідовність жорстка:
**спершу OCPP (Промпт 3) → потім витрата балансу/оплата за фактом на станції.**

Проста розетка без лічильника (як поточна Home, Schuko 7 кВт) енергетичний
білінг не вміє: не вимірює спожите й не зупиняється на N кВт·год. На таких —
лише фіксована ціна за сеанс.

## Поточний стан (2026-07-23)
- **Є (buy-side):** купівля пакетів (50 кВт·год / 750 грн, 100 / 1350 −10%),
  баланс у кВт (`users.balance`), ваучер, «Мій білінг» — **на ТЕСТОВОМУ**
  Telegram-Payments токені (`PAYMENT_PROVIDER_TOKEN`), тобто не за реальні гроші.
- **Немає (spend-side):** витрати балансу на станції, розрахунку за фактом
  споживання, вибору кВт/грн на QR-сторінці.

## Рішення по гаманцю
- **KWH_RATE_UAH = 15** грн/кВт·год — тимчасовий, єдина константа платформи,
  показується водію до підтвердження.
- **Вибір водія на станції:** баланс (куплені кВт) АБО грн за спожиті.
- **Ціна за фактом після сесії** — потребує метрингу (OCPP). Без нього — тільки
  передоплата/фікс.
- **Фаза 6a — гаманець лише на ВЛАСНИХ станціях eVolt** — юридично безпечно.
  **Фаза 6b — сторонні оператори** = Модель C (платіжне посередництво) =
  юридичний гейт (див. `billing-commission-models.md`).
- **Реальні продажі пакетів** робимо через **Monobank-еквайринг eVolt** (уже
  працює), а не через живий Telegram Payments. (Найближча задача — перехід
  з тестового на реальний mono.)

## OCPP (Промпт 3) — план
**Стек:** бібліотека `ocpp` (mobilityhouse), **OCPP 1.6J**, WebSocket-ендпоінт
у наявному FastAPI, розробка/тест проти **симулятора** (реальне залізо — потім).

**Інфраструктура:** nginx проксіює WebSocket на `/ocpp` (Upgrade/Connection) —
робиться вручну на сервері (як HTTPS), не в коді.

**Фазування (щоб рев'ю було реальним):**
- **3a — кістяк:** WS-ендпоінт `/ocpp/{cp_id}`, підключення, `BootNotification` /
  `Heartbeat` / `StatusNotification`, реєстр зарядок (перевикористати наявні
  `ocpp_charge_point_id` / `get_station_by_ocpp_charge_point_id`), OCPP basic
  auth, тест на симуляторі. Без транзакцій і оплати.
- **3b — транзакції + метринг:** `Authorize`, `Start/StopTransaction`,
  `MeterValues`, `RemoteStart/StopTransaction`, прив'язка до сесій.
- **3c — інтеграція оплати/гаманця:** дистанційний старт при оплаті/балансі;
  на `StopTransaction` відомі фактичні кВт·год → ціна після сесії → списання
  з kWh-балансу АБО грн через **Monobank hold+capture** (заблокувати наперед,
  списати фактичну після). Вибір кВт/грн. Це і є повна модель.

## Послідовність
1. **Зараз:** перехід продажу пакетів з тестового Telegram Payments на реальний
   **Monobank-еквайринг** (незалежно від OCPP).
2. **Промпт 3a → 3b → 3c** (OCPP + метринг + витрата на станції за фактом).
3. **6b (сторонній гаманець)** — після юридичного висновку.

## Відкриті питання
- Фінальне значення й керування KWH_RATE_UAH.
- Фіскалізація продажу пакетів (висновок бухгалтера) — це власний продаж eVolt,
  простіше за сторонній кейс.
- Модель реальної зарядки з OCPP/лічильником (для налаштування 3b/3c).
- Перевірити підтримку Monobank **hold+capture** (для грн-post-pay у 3c).
