# 🏷️ ЯДРО: White-Label Білінг для власників зарядних станцій

## Проблема і клієнт

Готель, ЖК, СТО, кафе чи мала мережа ставить 1–10 зарядних станцій — і впирається в софт: як брати гроші з чужих водіїв? Варіанти зараз погані: (а) віддати станцію під бренд великої мережі — втрата контролю і маржі; (б) enterprise-платформи рівня Monta/AMPECO — дорого і складно для 2 станцій; (в) "пишіть на Viber, скинете на карту" — так працює половина малих операторів України. 

**Ви продаєте лопати тим, хто вже копає:** клієнт уже вклав $1–5k у залізо і ХОЧЕ заробляти — ваш софт це вмикає. Це принципово легша продажа, ніж маркетплейс: не треба збирати дві сторони ринку, клієнт монетизується з дня 1.

**Сегменти (Україна → ЄС):** готелі/ресторани зі станціями; ОСББ і ЖК з паркінгами; СТО/детейлінги; малі мережі 3–15 станцій без свого софту; забудовники (станція = амеńitity). У Польщі/Чехії — тисячі таких мікро-операторів, ринок фрагментований, великі платформи ними не займаються.

## Продукт (MVP)

Водій сканує QR на станції → веб-сторінка (без установки застосунку) → бачить тариф оператора → платить карткою → станція вмикається (або оператор вмикає вручну на не-розумних станціях) → по завершенні чек у SMS/Telegram. Оператор у своєму кабінеті (Telegram-бот + веб): тарифи, транзакції, виручка, виплати на свій рахунок, статистика по портах.

**Три рівні підключення станцій:**
1. **Ручний** (будь-яка розетка/станція): оператор підтверджує старт, вводить kWh — нуль інтеграції, працює з першого дня
2. **OCPP 1.6J** (більшість розумних станцій): автостарт/стоп, реальний облік kWh — головна технічна робота MVP
3. **OCPI** — ваш готовий код: коли оператор захоче віддавати станції в роумінг великим eMSP — ви це вже вмієте, конкуренти-початківці ні

## Що перевикористовується з ev-charge-bot

| Є в коді | Стає в білінгу |
|---|---|
| Баланс + `kw_transactions` (єдина точка запису) | Гаманці операторів + журнал транзакцій по станціях |
| Monobank еквайринг + ECDSA-верифікація + reconcile | Прийом оплат водіїв + автозвірка (готова!) |
| Telegram-бот, FSM, планувальник | Кабінет оператора + алерти («станція офлайн», «оплата пройшла») |
| OCPI-модуль | Роумінг як преміум-фіча (актив у запасі працює!) |
| Тести, CI, health, бекапи | Та сама інженерна культура |

**Оцінка перевикористання: ~70–80%.** Нове: мультитенантність (оператори замість одного балансу водіїв), OCPP-сервер (websocket), QR-flow оплати без реєстрації, виплати операторам.

## Монетизація і юніт-економіка

- **Комісія платформи: 5% з обороту, без абонплати** (рішення 2026-07-23, на старт; попередня модель «підписка 590/1490/2990 грн/міс + 3–5%» скасована). Оплата йде напряму оператору через його еквайринг; 5% — плата за платформу.
- Приклад юніта: готель з 2 станціями, 60 сесій/міс × 300 грн = 18 000 грн обороту → ваші 5% = **~900 грн/міс ($22) з одного клієнта**. Без абонплати — нижчий бар'єр входу для оператора: ставка на обсяг і швидше підключення.
- ⚠️ Фінмодель нижче (MRR, таблиця росту) рахована під СТАРУ підписну модель — переглянути під «5% без абонплати» окремо.
- CAC при inbound-продажах — десятки доларів. Це найздоровіша юніт-економіка з усіх розглянутих проєктів

## Конкуренти і позиціювання

- **Monta, AMPECO, ChargeLab** (ЄС/США) — enterprise: онбординг тижнями, ціни від сотень євро, мінімуми по станціях. Мікро-оператора з 2 станціями вони не хочуть
- **Українські мережеві оператори** (TOKA, YASNO) — беруть станції "під себе", тобто конкурують з клієнтом, а не служать йому
- **Ваша ніша:** «білінг за 15 хвилин для 1–10 станцій, без договорів на рік, українською, з підтримкою в Telegram». Цінова парасолька під enterprise-гравцями величезна

## Продажі БЕЗ «полів» (соло-сумісні)

1. **Пілоти:** 3 оператори зі знайомих/EV-чатів безкоштовно на 2 міс за кейс з цифрами («готель Х заробив 19 тис. грн за місяць на 2 станціях»)
2. **Inbound-контент:** 2 статті («Скільки заробляє зарядка при готелі: реальні цифри», «Як монетизувати станцію в ЖК: інструкція») → SEO + посів у групах готельєрів/ОСББ/EV
3. **Продавці станцій — головний канал:** компанії, що продають/монтують зарядки, щодня чують «а як брати оплату?» — партнерка 20% від річної комісії з клієнта. 5–7 таких партнерів закривають потребу в продажах повністю
4. **Каталоги:** картки станцій операторів автоматично публікуються в PlugShare/OCM — клієнту трафік, вам вітрина

## Ризики і kill-критерій

- **OCPP-зоопарк:** станції різних вендорів по-різному тлумачать протокол — закладайте час на сумісність, стартуйте з ручного рівня
- **Платіжна модель:** гроші водіїв ідуть оператору напряму через його еквайринг (ви — софт) АБО через вас з виплатами (ви — платіжний посередник, юрконсультація обовʼязкова). MVP — перший варіант: простіше юридично, Monobank-еквайринг оператора підключається у вашому флоу
- **Kill-критерій:** < 5 платних операторів через 3 міс після пілотів → модель не тягне, перемикання на ФОП

## Промпти для Claude Code (конвенції ev-charge-bot)

*Перед стартом — виконати «Обовʼязкову підготовку» (розділ нижче в частині P2P: секрети в git, alembic, pytest) — вона спільна для обох ліній.*

**Стартовий:** як в існуючому проєкті (Python 3.11, asyncpg, Alembic + idempotent-дзеркало, гілка `feature/white-label` від `main`) + «мультитенантність: КОЖНА таблиця і КОЖЕН запит — з operator_id; тести на ізоляцію даних між операторами обовʼязкові».

**Промпт 1 — Тенанти і станції:**
> Міграція 0010: operators (name, phone, telegram_id, status, commission_pct DEFAULT 4, monobank_token_encrypted NULL), operator_stations (operator_id, name, address, lat/lng, connector_type, power_kw, mode: manual/ocpp, ocpp_charge_point_id NULL, tariff_uah_kwh, tariff_uah_start NULL, qr_slug UNIQUE, status), operator_sessions (station_id, started_at, ended_at, kwh, amount_uah, payment_id, status), operator_payout_ledger. Репозиторій у стилі проєкту + тести ізоляції тенантів (оператор А ніколи не бачить дані Б).

**Промпт 2 — QR-flow оплати водія:**
> Веб-сторінка /s/{qr_slug} (FastAPI + Jinja2, мобільний-перший): тариф і статус станції → «Розпочати» → створення Monobank-інвойсу (еквайринг ОПЕРАТОРА, його токен з operators, розшифрування через ENCRYPTION_KEY) → після webhook-підтвердження: manual-режим — пуш оператору «Оплачено 200 грн, увімкніть станцію» з кнопкою підтвердження; сесія пише в operator_sessions. Чек водію на сторінці + опційно в Telegram. Без реєстрації водія.

**Промпт 3 — OCPP-сервер:**
> Додай OCPP 1.6J Central System на бібліотеці `ocpp` (websocket-ендпоінт /ocpp/{charge_point_id}): BootNotification, Heartbeat, StatusNotification, Start/StopTransaction, MeterValues, RemoteStart/StopTransaction. Привʼязка charge_point_id → operator_stations. Після оплати водія — RemoteStartTransaction; StopTransaction → фінальні kWh в сесію. Симулятор станції для тестів (окремий скрипт), стани станції в реальному часі в кабінеті оператора.

**Промпт 4 — Кабінет оператора:**
> Розшир бота розділом «🏷️ Мій білінг»: онбординг оператора (назва, еквайринг-токен Monobank — шифруємо), майстер додавання станції з генерацією QR (PNG для друку), тарифи, сьогоднішні сесії, виручка за період, вивантаження CSV. Веб-версія кабінету — та сама логіка через FastAPI+Jinja2 з логіном за одноразовим кодом з бота.

**Промпт 5 — Звірка і алерти:**
> Розшир reconcile-механіку режимом --operators: сесії × тарифи == інвойси Monobank по кожному оператору; розбіжність → алерт. Алерти операторам: станція OCPP offline > 10 хв, невдала оплата, добовий підсумок о 21:00. Адмін-панель для вас: список операторів, MRR, комісії до виставлення (щомісячний інвойс оператору на комісію, поки без автосписання).

# 🦄 Єдиноріг-траєкторія: від білінгу до віртуальної електростанції

*Стратегічний горизонт 3–5 років. НЕ впливає на 90-денний план — але пояснює, чому кожен рядок коду білінгу цінніший, ніж здається.*

## Логіка сходинок

**Рік 1 — Білінг (зараз):** оператори + станції + OCPP-керування + облік kWh. Кожен платний оператор = керована точка мережі в майбутньому.

**Рік 2 — Smart Charging:** та сама команда RemoteStartTransaction, але за сигналом ціни/навантаження: «заряджай вночі, коли дешево», «пригальмуй у пік». Оператору — економія на електриці, вам — преміум-тариф і перші дані гнучкості. Українська специфіка (графіки відключень, генератори, сонце) = унікальна експертиза «зарядки при нестабільній мережі», якої немає в західних конкурентів.

**Рік 3+ — Агрегація гнучкості (VPP/V2G):** тисячі керованих зарядок = віртуальна електростанція. Продаж гнучкості операторам мереж на ринках ЄС; з V2G — авто віддає енергію в пік. Орієнтири масштабу цього шару: Kraken (Octopus), Virta, Monta — мільярдні оцінки. Сюди — вже з венчурними грошима і командою, під метрики років 1–2.

## Що це міняє СЬОГОДНІ (майже нічого — і в цьому сила)

- OCPP-модуль (Промпт 3) проєктувати з думкою «сесією можна керувати ззовні» — це і так закладено в протоколі
- Зберігати погодинні дані сесій (старт/стоп, kWh, потужність по MeterValues) — сировина для алгоритмів гнучкості; коштує кілька стовпчиків у БД
- У майбутніх пітчах інвесторам: «білінг — наш клин у ринок енергогнучкості», а не «софт для QR-оплат»
- Все інше — без змін: спершу 5 платних операторів, потім мрії

**Тригер сходинки 2:** 30+ операторів з OCPP-станціями і перший запит «а можна заряджати, коли дешевше?» — він прийде сам.

# 🗺️ Канал залучення: Карта-агрегатор станцій у Telegram

*Growth-механіка, не дохід. Місце в 90-денному плані: дні 19–24 паралельно з ФОП, або одразу після 3-го пілота білінгу. Не повинна відсунути перший платний оператор.*

## Ідея

Водій зараз живе в зоопарку застосунків (TOKA, YASNO, окремі мережі). Даємо одну безкоштовну живу карту всіх станцій України в Telegram: OCM-база (вже підключена в боті) + станції наших операторів. Водій приходить дивитись — залишається користувачем.

## Український кілер-фактор: краудсорсинг живого статусу

Кнопки під кожною станцією: «✅ працює · ⚡ є світло · 🚗 черга N авто · ❌ не працює» з таймстампом і ніком. У країні графіків відключень це цінніше за офіційні дані мереж — і цих даних немає ні в OCM, ні в PlugShare, ні в комерційних застосунках. Це наш унікальний датасет з першого дня.

**Важливо:** живі статуси чужих мереж через їхні API/скрейпінг — сіра зона; краудсорсинг — легальний і унікальніший шлях.

## Воронка

1. Карта (безкоштовно) → база водіїв росте
2. Станції білінг-операторів — зелена позначка «верифіковано · оплата в 1 клік» і пріоритет у видачі → карта ПРОДАЄ білінг операторам: «N водіїв щотижня шукають, де зарядитись — ваша станція буде з кнопкою оплати»
3. База водіїв = готовий попит для P2P-лінії потім
4. Розголос: пости «зробив живу карту зарядок з чергами» в EV-чатах розлітаються самі — це контент, який спільнота чекає

## Промпт для Claude Code

> Розшир існуючий OCM-пошук до карти-агрегатора: (1) Mini App екран карти (Leaflet + OpenStreetMap): піни OCM-станцій + operator_stations (зелений пін «верифіковано») + p2p_stations коли зʼявляться; кластеризація, фільтри: конектор, потужність, «⚡ є світло зараз», «✅ працювала < 24 год тому». (2) Таблиця station_reports (station_ref TEXT — ocm:{id} / op:{id}, user_id, status CHECK IN ('works','power_on','queue','broken'), queue_len SMALLINT NULL, created_at); індекс по station_ref+created_at. (3) У картці станції — 4 кнопки статусу + стрічка останніх 3 репортів («✅ працює · 40 хв тому»); агрегований бейдж на піні за останнім репортом < 24 год. (4) Анти-спам: 1 репорт на станцію на користувача за 2 год; user_id з 3+ «спірними» репортами (наступний репорт протилежний протягом 1 год) — flag адміну. (5) Легкий гейміфікатор: лічильник корисних репортів у профілі, топ-10 тижня в каналі. Кеш OCM-видачі в Redis 15 хв, щоб не впертись у ліміти.

## Метрика каналу

- 500 водіїв у боті за перші 4 тижні після посіву в EV-чатах
- ≥ 30 репортів статусу/тиждень (означає, що механіка жива)
- ≥ 2 оператори білінгу, які прийшли «з карти» за квартал — тоді канал працює як воронка, а не як іграшка

# 🔌 ДРУГА ЛІНІЯ: P2P-Зарядки (запуск після 10+ платних операторів білінгу)

*Оригінальний план нижче збережено повністю. Синергія: приватні власники станцій з P2P — це майбутні оператори білінгу, і навпаки; кодова база спільна.*

## Стратегія: гібридна мережа

ev-charge-bot уже вміє: користувачі, баланс у кВт·год з журналом (`kw_transactions`), Monobank-поповнення з ECDSA-верифікацією, пошук станцій (OCM), сесії через OCPI з CPO-партнером. **P2P — це друга категорія станцій у тому самому боті:**

| | Комерційні (є зараз) | Приватні P2P (додаємо) |
|---|---|---|
| Джерело | CPO через OCPI 2.2.1 | Власник додає сам через бота |
| Старт сесії | OCPI START command | Бронювання слота + ручне підтвердження власника |
| Оплата | Списання з балансу кВт·год | **Те саме списання з того ж балансу** |
| Виплата | Розрахунки з CPO | 90% власнику, 10% комісія (тижневі payouts) |

**Ключова перевага гібрида:** водію байдуже, чия станція — він бачить одну карту. А для бізнесу P2P знімає залежність від єдиного CPO-партнера і дає покриття там, де комерційних станцій немає (у Львові публічно працює ~28 станцій при 123 затверджених локаціях — дефіцит величезний).

**Головне архітектурне рішення:** водій платить за P2P-сесію з існуючого балансу кВт·год. Жодної нової платіжної інтеграції — Monobank-поповнення вже працює. Додається лише облік заробітку власників і виплати.

## Навіщо (і коли стартувати)

- **Тригер:** 300+ P2P-сесій/міс або angel-раунд — не раніше; до того бот + PWA дешевше валідують попит

- Довгостроково: CarPlay / Android Auto — станції прямо в бортовій навігації, бот такого не дасть ніколи

## Технологічний вибір

**React Native + Expo** (рекомендація для розробки AI-агентом однією людиною):

- Одна кодова база iOS + Android; величезна кількість прикладів у навчальних даних AI → якісна генерація
- Expo OTA-оновлення: правки JS без проходження рев'ю сторів
- `react-native-maps` (Google/Apple Maps), `expo-notifications` (пуші), WebView для Monobank checkout
- Альтернатива Flutter — теж валідна, але екосистема Expo зручніша для соло-оператора

**Backend не переписується.** FastAPI вже віддає HTTP — додається мобільний API-шар поверх існуючої логіки.

**Стартовий (backend, разом з PROJECT_CONTEXT.md):**
> Працюємо в існуючому ev-charge-bot, гілка feature/mobile-api від main. Конвенції проєкту: Python 3.11, asyncpg сирий SQL, Alembic + idempotent-дзеркало в create_tables(), баланс тільки через update_user_balance(). Створюємо мобільний API-шар app/api/mobile/ поверх ІСНУЮЧИХ сервісів (booking_service, commands_service, ocm_service) — бізнес-логіку не дублювати, тільки викликати. Всі ендпоінти під префіксом /api/mobile/v1, авторизація JWT (PyJWT): access 15 хв / refresh 30 днів, refresh-токени в новій таблиці (міграція). Тести в стилі проєкту — фейкові зʼєднання, без живої БД.

**Промпт 1 — Auth:**
> Реалізуй два способи входу: (а) телефон + SMS OTP (провайдера абстрагуй інтерфейсом, для дева — код у лог; продакшн-адаптер TurboSMS додамо пізніше); (б) через Telegram: ендпоінт створює одноразовий 6-значний код зі строком 5 хв, користувач вводить його боту (новий хендлер /link), бот привʼязує telegram_id до users і код гаситься. Обидва шляхи ведуть до одного users-запису за номером телефону. Видача/оновлення/відкликання JWT, rate-limit на OTP (5 спроб/год на номер).

**Промпт 2 — Stations + Bookings API:**
> GET /stations: обʼєднана видача комерційних (існуючий шар OCM/OCPI) і p2p_stations у радіусі, спільна модель відповіді (Pydantic, як у app/schemas/), поле source: commercial|p2p. GET /stations/{id}/slots — вільні слоти P2P. POST /bookings, DELETE /bookings/{id} — через існуючий booking_service (правила скасування вже там). POST /sessions/start|stop для комерційних — через commands_service з перевіркою балансу. Swagger /docs для мобільних ендпоінтів — закритий у продакшні (перевір існуючу конфігурацію).

**Промпт 3 — Balance + Push:**
> GET /balance (+ пагінована історія kw_transactions), POST /balance/topup → створення Monobank-інвойсу існуючим шляхом, повернути checkout URL. Пуші: таблиця device_tokens (міграція), POST /devices для реєстрації Expo-токена, push_service.py на expo-server-sdk з чергою через існуючий Redis; підключи відправку в точки подій: підтвердження/нагадування бронювання, старт/фініш сесії, успішне поповнення (Monobank webhook вже ловить — додай хук).

**Промпт 4 — Застосунок (окремий репозиторій):**
> Створи React Native + Expo (TypeScript) застосунок eVolt UA: навігація expo-router, стан TanStack Query + аксесори до /api/mobile/v1, зберігання токенів у expo-secure-store з авто-refresh. Екрани: Auth (телефон/OTP + кнопка «Увійти через Telegram» з deep-link), Map (react-native-maps, кластеризація, фільтри, геолокація), StationCard, Booking (вибір слота), ActiveSession (polling статусу), Balance (WebView Monobank з поверненням за deep-link), MyBookings, Profile. Пуші через expo-notifications (реєстрація токена після логіну). Темна тема за замовчуванням (водії — вечір/ніч). Мова UI — українська.

**Промпт 5 — Реліз:**

Чек-лист сабміту в обидва стори + тексти описів українською й англійською. Налаштуй OTA-канал production для JS-оновлень без ребілду.
