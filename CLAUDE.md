## Технічні конвенції (обовʼязкові, деталі — docs/PROJECT_CONTEXT.md)

- Python 3.11 · FastAPI · aiogram 3.x · asyncpg із **сирим SQL без ORM** · Alembic
- Єдині екземпляри `bot`/`dp` — з core loader; нових не створювати
- Кожна Alembic-міграція дублюється idempotent-блоком у `create_tables()`/`init_operator_tables()` — оновлювати ОБИДВА місця (тестується)
- Будь-яка зміна балансу користувача — ТІЛЬКИ через `update_user_balance()`; ніколи напряму в баланс чи журнал транзакцій
- Мультитенантність білінгу: кожна таблиця і кожен запит — з `operator_id`; тести ізоляції тенантів обовʼязкові (`test_operator_isolation.py`, `TENANT_SCOPED_CALLS`)
- Будь-який текст оператора з `parse_mode="HTML"` — через `html.escape()`
- Monobank-вебхук: не довіряти тілу — re-verify через `get_invoice_status` токеном оператора; невідомий invoiceId → тихий 200
- Ідемпотентність оплат: партк. unique-індекс `uq_ledger_session_income` + mutex `UPDATE ... WHERE status <> ...` — перевикористовувати, не переписувати
- `ENCRYPTION_KEY` (Fernet) шифрує токен оператора в БД — НЕ міняти без перешифрування
- Секрети — тільки `.env` (НЕ в git); ніколи не комітити, не логувати
- Тести — на фейкових зʼєднаннях без живої БД, у стилі існуючих; лишаються зеленими
- Прод (DigitalOcean droplet) агентом напряму не чіпати: локальна розробка → git push → деплой руками

## Правила виконання задач

1. Один промпт = один модуль; після модуля — інструкція запуску + тести. Не генерувати «все одразу»; великі фічі (OCPP) різати на під-бандли (3a/3b/3c)
2. Платіжна логіка, міграції, баланси — пояснюй план ДО написання коду і познач місця для ручного рев'ю; результат іде на рев'ю Opus 4.8 як `review_promptN.md`
3. Якщо задача суперечить конвенціям вище — зупинись і спитай, а не обходь
4. Дані/цифри в бізнес-файлах — оцінки; не сприймай як факти про ринок без перевірки

## Бізнес-контекст одним абзацом

Соло-засновник, повний фокус, бюджет ~0 → шлях: пілоти → перші платні оператори → реінвест.
